## 开发规则

- 只实现当前任务所需的最小代码
- 不做无关修改或大规模重构（除非明确要求）
- 优先复用已有代码，不重复实现
- 保持代码清晰、模块化、可测试
- 不破坏现有功能（有风险需提前说明）

## AI行为

- 不要假设需求，信息不足先提问
- 基于现有代码修改，避免新建重复实现
- 输出前先简要说明思路
- 避免过度设计
- 涉及环境/配置（DB / Supabase / AWS）必须明确说明需要我做什么

## 输出规则

- 每次回复称呼我为【编酱】
- 输出结构：
  1. 先说明思路（简洁）
  2. 再给代码
  3. 必要时说明修改点
- 对话结束时进行复盘：
  总结刚才有哪些理解偏差，以及如何优化我的描述

## Git版本管理

- 每完成一个独立功能，进行一次 commit
- 不要将多个无关修改合并在一个 commit

commit 规范：

- feat: 新功能
- fix: 修复问题
- refactor: 重构
- style: UI/样式调整
- docs: 文档
- chore: 杂项

## 注释规范

- 不删除已有注释
- 不添加无意义注释（避免解释“做了什么”，优先说明“为什么这样做”）

必须写注释的地方：

1. 核心业务逻辑（如六爻计算规则）
2. 复杂 UI 结构（如六爻排盘 grid）
3. 不直观的实现（特殊判断 / hack）

注释要求：

- 简洁清晰
- 优先解释“为什么”

## 项目文档（补充）

文档必须与代码同步更新。**正式文档仅为仓库根目录三份**，互相引用即可，**不单独维护 `docs/` 目录**。

- README.md：项目简介、技术栈、本地运行、指向 PRD / architecture 的索引
- PRD.md：产品定位、核心功能、用户流程
- architecture.md：技术选型、项目结构、核心模块、**前后端接口与数据约定**、数据模型、关键设计决策

根目录其他 Markdown（如历史草稿）仅供个人对照，**不作为正式引用来源**。
