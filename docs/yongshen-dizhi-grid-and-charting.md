# 用神结果趋势：月支 × 日支（144）与图表化（规划）

> 背景：在固定卦盘与固定用神爻位下，**月地支、日地支**变化会带来 12×12 = **144** 种离散组合；希望除「当前时刻」单点结果外，还能用**图表**展示指标随组合变化的趋势。  
> 关联：[PRD.md](../PRD.md) 用神与后续迭代 · [architecture.md](../architecture.md) 爻位与接口约定 · [.cursor/plans/用神选择与交互.plan.md](../.cursor/plans/用神选择与交互.plan.md) 页面分区。

---

## 1. 规划层面：把「144」当成一个有限离散空间

- **数学上**：不是时间序列，而是 **12×12 网格上的标量场**（每个格子一个用神强弱或分数）。图表可选：
  - **热力图（Heatmap）**：横轴月支、纵轴日支，颜色映射数值；一眼看全局。
  - **曲线图**：固定日支扫月支（12 点）或固定月支扫日支（12 点），适合叙事「随月变化」。
  - **当前点标注**：在热力图上高亮「当下月日」所在格，与单点结果一致。
- **产品上**：默认仍展示**当前单点**；「144 全盘」放在 **折叠高级区**（与 PRD 中纵向分区一致），避免首屏过载。
- **交互**：允许用户点某一格 → 展示该格详细说明或中间量（若后端返回）。

---

## 2. 后端与契约：不要用浏览器打 144 次 HTTP

| 方案 | 说明 |
|------|------|
| **不推荐** | 前端 `for` 循环 144 次 `GET countYongshen`：延迟高、易被限流、难缓存、难观测。 |
| **推荐** | **单次批量接口**：例如 `POST` 或 `GET`，body/query 含 `liuyao_id`、`yongshen`（爻位 1–6），返回 **长度 144 的数组**（或 12×12 嵌套），元素含 `month_zhi`、`day_zhi`、`value`（及可选中间特征）。 |
| **可选优化** | 服务端对同一 `(liuyao_id, yongshen)` 做**短期缓存**（盘不变则 144 向量不变）；版本号随排盘规则变而失效。 |

**响应形状示例（示意，定稿时写入 architecture / 草稿）**：

```json
{
  "liuyao_id": 1,
  "yongshen": 3,
  "points": [
    { "month_zhi": "子", "day_zhi": "子", "value": 0.42 },
    { "month_zhi": "子", "day_zhi": "丑", "value": 0.38 }
  ]
}
```

顺序约定需在契约中写死（例如先固定月支、内层扫日支），前后端同一套枚举顺序。

## 3. 计算实现（Java / 手搓逻辑侧）

- **循环 144 在服务端一次完成**：单次请求内双层循环，复用已装好的卦象结构，只替换「月日」参与旺衰的输入，避免 144 次重复 IO。
- **中间结果结构化**：若未来要做误差分析，建议每条格子或抽样格子带 **debug 字段**（可选 query `?debug=1`），便于对照手算。
- **与现有 `countYongshen` 关系**：批量接口内部可循环调用现有纯函数；稳定后再内联优化，避免过早复制逻辑两份。

---

## 4. 前端写代码时的建议（Next + 现有 UI 规则）

1. **数据获取**：Client Component 内 `useEffect` 或按钮「生成 144 热力图」触发 **一次** fetch；`loading` / `error` 明确。
2. **图表库**：选一种即可（如 Recharts、ECharts for React、visx）；热力图若库支持弱，可用 **CSS grid + 背景色** 自绘 12×12（与 `.cursor/rules/UI.mdc` 的 grid 优先一致），**移动端允许横向滚动**承载色条与坐标轴。
3. **性能**：144 个点体量小；避免在 `mousemove` 上重算全量；tooltip 读缓存即可。
4. **无障碍**：色觉依赖时辅以数字文本或图例分级；重要结论仍用一句话摘要（不只靠颜色）。
5. **与结果页布局**：热力图 / 图放在 **用神结果区下方**，不替换六爻排盘主 grid。

## 5. 分阶段落地（建议里程碑）

| 阶段 | 内容 |
|------|------|
| M0 | 单点：当前月日 + 现有 `countYongshen`（已有）。 |
| M1 | 后端批量 144 + 契约；前端表格或简易 12×12 色块。 |
| M2 | 热力图 + 当前点高亮 + 折叠高级区入口。 |
| M3 | 点格详情、debug 开关、缓存与观测日志（与 [offline-evaluation-and-algorithm-iteration.md](./offline-evaluation-and-algorithm-iteration.md) 衔接）。 |

---

## 6. 风险与注意

- **业务含义**：144 格是「模型在给定规则下的相对强弱」，需在 UI 文案中避免被理解成绝对预言。
- **鉴权与成本**：批量接口若计算较重，需鉴权、限流或与会员策略对齐（产品决策）。
