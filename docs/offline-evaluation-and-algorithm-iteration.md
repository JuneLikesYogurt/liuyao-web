# 离线评估与算法迭代（远期存档）

> 目的：卦例增多后，若「用神计算结果 / 趋势」与经验或标签不符，如何用数据驱动方式**定位问题**并**迭代规则与系数**，而不必一开始就上黑盒深度学习。  
> 与产品功能文档关系：本文件为**工程与方法论备忘**；接口与页面仍以 [PRD.md](../PRD.md)、[architecture.md](../architecture.md) 为准。

---

## 1. 问题分层（先分清「错在哪」）

| 类型 | 含义 | 典型对策 |
|------|------|----------|
| 标签噪声 | 「应验 / 吉凶」记录不准、口径不一、事后偏差 | 收紧标签定义、多源校验、只用高置信子集做校准 |
| 特征不足 | 盘内信息未进公式（伏神、月破、回头克等） | 补规则与中间特征，再谈调系数 |
| 系数 / 结构 | 公式形态大致对，权重或分支不对 | 系数表化、离线搜索或贝叶斯优化 |

---

## 2. 数据：可复现比「只有一个最终数」更重要

建议每条卦例（或可从 `liuyao_id` 重放）长期保留或可推导：

- **输入快照**：四柱、旬空、本卦 / 变卦、动爻、用神爻位、计算时语境（如月日干支若参与）。
- **输出快照**：不仅最终标量，尽量保留 **中间量**（若 Java 多步加权，结构化 JSON 或日志），便于事后分析「哪一步算反」。
- **版本**：`rule_version` / `coefficient_set_id`，避免无法区分「数据变了还是代码变了」。

---

## 3. 分析路径（从简到难）

1. **描述性统计与分层**：按六亲、动静、旬空、卦宫等切片看误差是否集中（指向缺分支而非全局调一个系数）。
2. **时间 / 批次切分**：旧数据调参，新时间段验证，减轻录入风格与单一来源过拟合。
3. **模型角色**：优先用 **系数搜索 / 规则补丁**；若需 ML，倾向 **逻辑回归、树模型** 等作**校准层**或残差拟合，与可解释规则并存，而非立刻端到端替换。

---

## 4. 工程闭环（避免分析与线上脱节）

- **影子模式（Shadow）**：新版本与线上并行计算，只记差异，用户侧仍用旧版，直到离线指标达标。
- **回归集**：固定一批「金标准」卦例，每次改 Java / 系数必跑，防回归。
- **A/B 或功能开关**：按用户或卦例 hash 分流（若未来有业务指标需求）。

---

## 5. 标签（真值）口径

在动手调参前，应用一页纸固定：**什么算应验**、争议卦处理、时间窗、分类还是打分。否则数据越多，噪声越大，分析结论越不可信。
