## 2. 后端功能模块（按域）

### 2.1 认证 `/auth`（匿名可访问）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/auth/register` | POST | 注册：`username` + `password`（见 DTO），成功后返回 JWT |
| `/auth/login` | POST | 登录：单字段 **`identifier`** + `password`；`identifier` 可为 **用户名或邮箱**（同一框）。兼容旧 JSON：仍可使用键名 **`username`** 代替 `identifier`（Jackson `@JsonAlias`） |

**响应体（成功）** `AuthResponse`：

- `token`：JWT 字符串（subject 为 `userId` 的十进制字符串）
- `tokenType`：固定 `"Bearer"`
- `userId`：整数，与库表 `users.user_id` 一致

**错误**：`400`（注册业务错误）或 `401`（登录失败），body 形如 `{"error":"..."}`。

**登录解析规则（后端）**：对 `identifier` 做 trim；若包含字符 **`@`**，按 **邮箱** 查用户；否则按 **用户名** 查。密码与 `password_hash` 做 BCrypt 比对。

### 2.2 起卦（排盘入库）`POST /`

- **路径**：根路径 `POST http://127.0.0.1:8080/?title=...&date=...&result=...`（查询参数，非 JSON body）
- **鉴权**：Spring Security 要求 **`POST /` 已认证**（需有效 JWT，见下节 **重要缺口**）
- **行为**：根据摇卦结果串、日期、标题等写入 `liuyao` 等，并返回 **`liuyao_id`（整数）**；响应可能是纯数字文本或 JSON，前端代理已做解析

**Principal**：控制器里 `Authentication.getPrincipal()` 为 **`Integer userId`**（来自 JWT）。

### 2.3 卦象结果详情 `GET /result`

- **路径**：`GET /result?liuyao_id=<id>`
- **鉴权**：**匿名允许**（`SecurityConfig` 中除 `POST /` 与 `/history` 外 `permitAll`）
- **响应**：`LiuYaoResultDetailDto`（JSON），含本卦/变卦、空亡、动静、干支等；详见下文「字段与爻位约定」

### 2.4 用神计数（可选）`GET /result/countYongshen`

- **参数**：`liuyao_id`、`yongshen`（爻位编号，与后端 `SuanGuaService` 约定一致）
- **返回**：`double`

### 2.5 历史列表 `GET /history`

- **鉴权**：**必须登录**（JWT）
- **参数**：`page`（默认 0）、`size`（默认 20，后端限制 1～100）
- **排序**：按 `date` **降序**
- **响应**：Spring `Page<LiuYaoHistoryItemDto>` 的 JSON 序列化（含 `content`、`totalElements` 等分页字段）；单项含 `liuyao_id`、`title`、`date`（东八区格式 `yyyy-MM-dd HH:mm:ss`）

## 3. JWT 使用方式

- **请求头**：`Authorization: Bearer <token>`
- **过滤器**：`JwtAuthenticationFilter` 解析 Bearer，将 **`userId`** 放入 `SecurityContext`（无角色细分时可为 `ROLE_USER`）

---

## 4. 前端现状与后端契约差异（Agent 必读）

### 4.1 起卦代理 **未带 Token**（与后端安全规则冲突）

- 后端：`SecurityConfig` 规定 **`POST /` 需要 authenticated**。
- 前端：`app/api/cast/route.ts` 使用 `fetch(BASE_URL)` **未设置 `Authorization`**。
- **后果**：若后端按当前代码部署，匿名起卦会 **401**；若你本地仍能起卦，可能是后端规则曾改动或未重启。
- **完善方向**：用户登录后将 `token` 存于安全位置（如 memory / HttpOnly cookie 需配合），由 **服务端 Route** 或 **仅服务端可调用的 API** 在转发时附加 `Authorization: Bearer ...`。

### 4.2 结果查询路径

- `lib/api.ts` 中 `getLiuYaoDetail` **直连** `BASE_URL/result?...`（Server Component 侧无 CORS 问题）。
- 仓库另有 `app/api/result/route.ts` 代理，**当前结果页未使用**；可统一为同源代理 + 环境变量 base URL。

### 4.3 历史页

- `app/history/page.tsx` 仍为 **静态占位**，未调用 `GET /history`，也未带 JWT。

### 4.4 登录 UI

- 若产品为「用户名或邮箱 + 密码」，提交登录时应使用 **`identifier`**（或兼容 **`username`**）与 `password`，与后端 `LoginRequest` 一致。

## 5. 数据与 UI：`gua_id` 与爻下标（避免画错卦）

后端 `GuaDetailDto` / 前端 `GuaInfo` 约定：

- **`gua_id`**：六位阴阳串，**索引 0 = 上爻，索引 5 = 初爻**（自上而下存储习惯）。
- **`yao_zhi` / `yao_liuqin` / `bengua_liushou_by_yao`**：**索引 0 = 初爻，索引 5 = 上爻**（与 `gua_id` 顺序相反）。

前端若用同一 `index` 驱动六亲、六兽列表，应对 **`gua_id`** 使用 **`5 - index`** 取字符，与 `lib/api.ts` 注释一致。

---

## 6. 用户表与 JPA（后端对齐摘要）

- 主键列名：**`user_id`**（Java 实体属性仍为 `id`）。
- 登录用邮箱依赖列 **`email`**（可空；非空时建议库侧唯一索引）。
- 密码仅存 **`password_hash`**（BCrypt）。

---

## 7. 建议后续（产品/工程）

- 用 **`NEXT_PUBLIC_API_BASE_URL` / `API_BASE_URL`** 统一替换硬编码 `127.0.0.1:8080`。
- 注册若需收集邮箱，需扩展 `RegisterRequest` 与前端表单，并保证邮箱写入后与登录 `identifier` 规则一致。
- 历史列表分页：前端解析 Spring Data `Page` JSON 的 `content` 数组。
